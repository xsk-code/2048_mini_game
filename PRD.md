# 2048 小游戏 PRD 需求文档

## 1. 项目概述

### 1.1 项目背景

2048 是由意大利程序员 **Gabriele Cirulli** 于2014年3月9日开发的益智数字拼图游戏。该游戏参考了《1024!》和《Threes!》的游戏模式，以其简洁的规则和深度的策略性迅速风靡全球。游戏于2014年3月9日开源发布在GitHub上，随后在同月19日被Ketchapp移植到iOS平台，引发了现象级的传播效应。

**信息来源**：[快懂百科](https://m.baike.com/wiki/2048/20024162)、[HandWiki](https://handwiki.org/wiki/Software:2048_(video_game))

### 1.2 项目目标

开发一款具有高度可玩性和优秀用户体验的经典2048小游戏，支持Web端运行，融合现代设计美学，打造令人印象深刻的视觉效果和流畅的交互体验。

### 1.3 开源项目参考

| 项目名称 | 技术栈 | 仓库链接 | 特点 |
|---------|--------|----------|------|
| Twenty_Forty_Eight | Python + tkinter | [GitHub](https://github.com/MELuisCruz/Twenty_Forty_Eight) | 桌面版，OOP实现 |
| 2048-pygame | Python + pygame | [GitHub](https://github.com/rajitbanerjee/2048-pygame) | pygame实现，经典复刻 |

## 2. 游戏规则

### 2.1 核心玩法

2048在一个 **4×4** 的网格棋盘上进行，玩家通过滑动数字方块来合并相同数字，最终目标是将数字合并到 **2048**。

### 2.2 详细规则

1. **初始状态**：游戏开始时，棋盘上随机出现 **2个** 数字方块（数值为2或4）
2. **移动操作**：玩家使用方向键（↑↓←→）或滑动手势，所有方块向该方向滑动
3. **合并机制**：
   - 当两个相同数字的方块相撞时，合并为一个方块，数值 = 两数之和
   - 合并后的方块在本次移动中不能再合并
   - 合并优先级：沿移动方向，靠近边缘的相同方块先合并
   - **三连合并规则**：若沿移动方向有三个相同数字连在一起，只有前两个合并
4. **生成规则**：每次**有效移动**后，在空余位置随机生成一个新的数字方块（90%概率为2，10%概率为4）
5. **无效移动规则**：移动后棋盘无任何变化时，**不生成新方块，不计分**
6. **胜利条件**：当棋盘上出现 **2048** 方块时，胜利（可选择继续游戏追求更高分）
7. **失败条件**：当棋盘填满且无法合并时，游戏结束

**信息来源**：[2048 Game Online](https://2048-gameonline.github.io/)、[Play-2048](https://play-2048.io/)

### 2.3 计分规则

- 每次成功合并方块，获得的分数 = 合并后方块的数值
- 最高分记录保存在本地

### 3.1 设计流程

**⚠️ 重要声明**：UI/UX设计必须调用 **frontend-design Skill** 进行设计，以确保设计质量达到生产级别标准。具体步骤如下：

3. 设计方案需要经过 skill 审核后，再进入开发阶段

### 3.2 视觉风格要求

根据 frontend-design skill 指导，设计方向建议如下：

- **风格定位**：现代极简 + 活力感，融合经典2048的温暖色调与当代设计美学
- **配色方案**：采用2的幂次方递进配色（参考经典2048配色），但需遵循现代色彩理论（推荐使用 OKLCH 色彩空间）
- **字体选择**：使用 distinctive 的显示字体，配合 refined 的正文字体
- **动效设计**：方块滑动、合并、出现需要有流畅的过渡动画

### 3.5 动画参数规范

| 动画类型 | 时长 | 缓动函数 | 描述 |
|---------|------|---------|------|
| 方块滑动 | 150ms | ease-out-quart | 方块从一个位置移动到另一个位置 |
| 方块合并 | 100ms | ease-out-quart | 两个方块合并为一个方块 |
| 方块出现 | 150ms | ease-out-back | 新方块从scale 0→1弹出 |
| 分数变化 | 200ms | ease-out | 分数数字变化时的跳动效果 |

### 3.6 触摸交互规范

- **滑动方向判定阈值**：手指滑动距离 ≥ 30px 触发方向判定
- **滑动方向判定**：取滑动距离较大的轴向作为滑动方向
- **防止误触**：仅在手指移动距离超过阈值后才判定为滑动操作

### 3.7 响应式设计

- **桌面端**：棋盘居中显示，键盘控制，最大宽度 500px
- **移动端**：棋盘自适应屏幕宽度（90vw，最大 400px），触摸滑动控制
- **断点**：
  - Mobile: < 640px
  - Tablet: 640px - 1024px
  - Desktop: > 1024px

### 3.8 UI设计规范（双主题：马卡龙 & 暗夜）

#### 设计理念

**美学定位**：柔和少女感（马卡龙） × 神秘高级感（暗夜）

**核心理念**：提供两套截然不同的视觉体验，让用户自由切换。"马卡龙"配色柔和温暖、充满少女感；"暗夜"配色深邃神秘、尽显高级质感。用户可以根据心情或场景自由选择，主题偏好会自动保存。

**差异化记忆点**：一键切换的双生美学 + 流畅的过渡动画

### 7.3 数据模型定义

```typescript
// 位置坐标
interface Position {
  row: number;    // 0-3
  col: number;    // 0-3
}

// 单个方块
interface Tile {
  id: string;           // 唯一标识（用于动画追踪）
  value: number;        // 方块数值（2, 4, 8, 16...）
  position: Position;   // 当前位置
  previousPosition?: Position;  // 移动前的位置（用于动画）
  mergedFrom?: Tile[];  // 合并来源（用于动画）
  isNew?: boolean;      // 是否是新生成的（用于动画）
}

// 棋盘状态
type Board = (Tile | null)[][];  // 4x4数组

// 游戏状态
type GameStatus = 'idle' | 'playing' | 'won' | 'lost';

// 游戏结果
interface GameResult {
  score: number;
  moves: number;
  status: 'won' | 'lost';
}

// 移动方向
type Direction = 'up' | 'down' | 'left' | 'right';

// 游戏状态（完整）
interface GameState {
  board: Board;
  score: number;
  bestScore: number;
  status: GameStatus;
  moves: number;
}
```

### 7.4 核心算法说明

| 算法 | 输入 | 输出 | 说明 |
|-----|------|------|------|
| `move(board, direction)` | 棋盘 + 方向 | { newBoard, score, moved } | 执行移动，返回新棋盘和得分 |
| `canMove(board)` | 棋盘 | boolean | 判断是否还有可用移动 |
| `addRandomTile(board)` | 棋盘 | 新的棋盘 | 在空位随机添加2或4 |
