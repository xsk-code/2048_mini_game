# 多棋盘主页与分模式存档设计

**日期：** 2026-04-20

## 目标

为当前微信小游戏增加一个主页，让用户进入游戏后先选择棋盘模式，再进入对应对局。

本次设计覆盖以下目标：

- 进入小游戏先显示主页，而不是直接开始 `4x4`
- 提供 `4x4`、`5x5`、`6x6` 三种棋盘模式
- 每种模式有独立的目标方块
- 每种模式有独立最高分
- 每种模式有独立存档，可在回到主页后继续
- 游戏内任意时刻可返回主页重新选择模式
- 主页采用“卡片式模式首页”结构，视觉方向为“甜点杂志感”

## 不在本次范围内

- 不增加联网能力
- 不引入新的外部依赖
- 不做整仓级重构
- 不增加复杂动画系统或完整自动化 UI 测试
- 不为到达目标方块增加胜利弹层

## 当前问题

当前项目只有单一入口，启动后直接进入 `4x4` 对局。核心逻辑和画布渲染都集中在 [game.js](/Users/sankan/Coding/SoloCoder/2048minigame/game.js) 中。

这导致几个问题：

- 没有模式选择入口
- `GRID_SIZE` 被写死为 `4`
- 最高分只有一个全局值，无法按模式区分
- 没有分模式存档
- 返回主页与切换模式的产品路径不存在

## 设计原则

- 优先小改，保留当前 canvas 渲染方式
- 让产品结构更完整，但不推倒重来
- 将“模式配置、棋盘规则、存档键名”从渲染逻辑中抽离
- UI 上突出模式选择，但不牺牲恢复进度的效率
- 尽量让不同棋盘尺寸共用同一套移动与结束条件逻辑

## 方案选择

采用“单画布多场景状态机”方案，并吸收少量模块拆分做法。

理由：

- 当前项目已经是单 canvas 结构，延续这一点改动最集中
- 主页、对局、结束覆盖层都可以用同一套渲染体系完成
- 相比彻底页面化或大拆分，更符合当前仓库的体量
- 同时通过抽出纯逻辑模块，避免 `game.js` 继续无限膨胀

## 产品与交互设计

### 主页

用户进入小游戏后先看到主页。

主页由三张模式卡组成：

- `4x4`
- `5x5`
- `6x6`

每张卡展示：

- 模式名称
- 目标方块
- 该模式最高分
- 是否存在可继续的存档

主页视觉方向：

- 保持当前项目的柔和主题基调
- 使用更精致的标题、层次和卡片装饰

### 进入模式

点击模式卡后进入该模式：

- 如果该模式存在存档，则恢复该模式当前进度
- 如果该模式没有存档，则创建新对局

本次不增加“继续 / 新开”二次确认弹层。卡片点击直接进入最合理的状态。

### 游戏中返回主页

游戏界面增加“回主页”按钮。

用户点击后：

- 当前模式进度立即保存
- 返回主页
- 主页上的对应模式卡应显示该模式可继续

### 切换模式

用户可在任意时刻返回主页，点击另一种模式。

切换时不弹确认提示，因为：

- 每个模式独立保存
- 切走不会丢局
- 返回后可以继续之前的模式

## 模式与规则设计

### 模式配置

新增一个模式配置表，至少包含：

- `id`
- `label`
- `gridSize`
- `targetTile`
- `bestScoreStorageKey`
- `saveStateStorageKey`

三种模式固定为：

- `4x4 -> 2048`
- `5x5 -> 4096`
- `6x6 -> 8192`

### 结束条件

所有模式统一使用同一条结束规则：

- 只有当棋盘没有空位，且不存在可合并的相邻方块时，才结束

这条规则不因为模式大小而变化，变化的只有棋盘尺寸与目标方块。

### 目标方块

目标方块用于：

- 表达模式难度
- 在主页卡片中展示
- 帮助用户理解不同模式的挑战等级

目标方块不用于强制结束游戏，也不触发胜利弹层。到达目标后玩家继续游戏。

## 数据设计

### 最高分

每个模式独立维护最高分：

- `4x4` 的最高分不影响 `5x5`
- `5x5` 的最高分不影响 `6x6`
- 主页展示各自的最高分

### 存档

每个模式独立维护一份对局存档。

存档至少包含：

- 当前模式 `id`
- 当前棋盘数据
- 当前得分
- 当前是否结束
- 当前方块自增 id 计数

当用户返回主页或完成有效移动后，应把当前模式状态写回对应存档。

当某模式没有可恢复内容时，应视为无存档，而不是恢复一个空壳状态。

## 视觉设计方向

视觉预期：

- 模式卡有明确层次和主次节奏
- 标题与副标题更有“封面感”
- 颜色沿用项目的马卡龙/夜晚基调，但不做普通卡片堆叠
- 不使用廉价玻璃拟态、通用紫蓝渐变或模板化 AI 样式

## 测试设计

本次测试以纯逻辑和状态数据为主。

至少覆盖：

- 不同棋盘尺寸下的结束条件
- 模式配置中的目标方块是否正确
- 三档模式的最高分键是否互相隔离
- 三档模式的存档键是否互相隔离
- 恢复模式时是否读取对应模式的棋盘和分数

UI 部分只做最小人工验证：

- 首次进入显示主页
- 点击模式卡可进入对应模式
- 游戏内可返回主页
- 返回主页后卡片能反映是否存在存档
- 再次进入能恢复对应模式

## 风险与控制

### 风险 1：`game.js` 继续膨胀

控制方式：

- 把模式配置和存档逻辑单独抽出
- 保持主文件只做场景、渲染和交互协调

### 风险 2：不同棋盘尺寸导致布局失衡

控制方式：

- 所有尺寸计算基于当前模式的 `gridSize`
- 字号和格子尺寸继续使用现有自适应策略
- 优先保证 `6x6` 在手机上仍可读可点

### 风险 3：模式切换时数据串台

控制方式：

- 所有分数和存档都使用带模式前缀的独立 key
- 测试中显式验证三档模式隔离

## 实施边界

本次实现应聚焦这四件事：

- 做出模式主页
- 接入三档棋盘配置
- 完成分模式最高分
- 完成分模式存档恢复

除此之外，不顺手扩展其他玩法或重构无关区域。
